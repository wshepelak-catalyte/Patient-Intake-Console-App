Patient Intake Console App
Project Background
Super Health Inc. wants a proof-of-concept console application for managing patient records before any web front end or database work begins. This phase of the project is scoped specifically for developers who are still learning Python — there is no HTTP API, no persistent database, and no authentication. All data lives in memory for the lifetime of the running program.
The goal of this phase is to practice:
* Designing clean, Pythonic data models using dataclasses and Enum
* Writing regex-based validation for structured text input
* Handling nested/composite objects (an address isn't just a string — it's its own type with its own rules)
* Building a simple menu-driven console interface
* Collecting all validation errors at once rather than stopping at the first bad field
________________


Functional Requirements
Patient Data Model
1. Given a patient's location is represented, then it is modeled as a dedicated Address type containing street, city, state, and zip_code fields, rather than as loose strings directly on the Patient type.
2. Given a patient's gender is represented, then it is modeled using an Enum type named Gender with exactly the members MALE, FEMALE, and OTHER.
3. Given the Address and Patient types are implemented, then both are defined using @dataclass, and every field on both classes has a type hint.
4. Given the height and weight fields are optional, then both are typed as Optional[int] and default to None when not provided.
5. Given a Patient instance is constructed, then no validation logic runs inside Patient.__init__ or __post_init__. Validation happens against the raw user input before a Patient object is created, so the model itself stays simple and the validation logic can be tested independently of the model.
6. Given a Patient instance exists, then it exposes a __str__ method (or a display() method) that returns a human-readable summary of all of its fields, including the nested address.
7. Given the application is running, then all created patients are held in a single in-memory collection (a list, or a dict keyed by an auto-generated id) — there is no external database in this phase.
Menu / Navigation
8. Given the program starts, then the user is shown a menu with options to list patients, create a patient, view patient details, edit a patient, delete a patient, and exit.
9. Given the user selects an invalid menu option, then an error message is shown and the menu is displayed again without crashing the program.
10. Given the user selects "exit," then the program terminates cleanly.
List Patients
11. Given the user selects "list patients," then every patient currently in memory is printed, showing only name, age, and gender for each.
12. Given no patients exist yet, then a friendly message is shown instead of an empty list.
View Patient Details
13. Given the user selects "view details" and provides a valid patient id, then all of that patient's fields are printed, including the full address.
14. Given the user provides an id that doesn't exist, then a clear error message is shown and the user is returned to the menu (no crash).
Create Patient
15. Given the user selects "create a patient," then the program prompts, one at a time, for: first name, last name, ssn, email, street, city, state, zip code, age, height, weight, gender, and insurance.
16. Given the user has finished entering all fields, then all fields are validated together (see Validation Requirements below) — the program does not validate and re-prompt field-by-field.
17. Given one or more fields are invalid, then the program prints every validation error at once (not just the first one found), and returns the user to the main menu without creating a patient. The user starts over from scratch on the next "create" attempt (no need to preserve partial input for this phase).
18. Given all fields are valid, then a Patient object is constructed, assigned a unique id, added to the in-memory collection, and the newly created patient's details are printed as confirmation.
Edit Patient
19. Given the user selects "edit a patient" and provides a valid id, then the program shows the patient's current values and prompts for new values for every field (student's choice: require re-entering everything, or allow blank input to keep the current value — document whichever approach you pick in your README).
20. Given the edited values are validated together and any are invalid, then all validation errors are shown at once and the original patient record is left unchanged.
21. Given the edited values are all valid, then the patient's record is updated in place and the updated details are printed.
Delete Patient
22. Given the user selects "delete a patient" and provides a valid id, then that patient is removed from the in-memory collection and a confirmation message is shown.
23. Given the user provides an id that doesn't exist, then a clear error message is shown (no crash).

Validation Requirements
Validate everything at once. Collect the raw values for all fields first, run every field through its validator, gather all the resulting error messages into a list, and only then decide whether to proceed (no errors) or report back to the user (one or more errors). Do not short-circuit on the first invalid field.
Use Python's re module for pattern matching. A clean way to structure this: one small validator function per field (or per type, for the nested Address), each returning either None (valid) or an error string. A validate_patient(raw_input: dict) -> list[str] function then calls each validator and collects the non-None results.
Every field is required (1 character minimum) unless marked optional below. A value that is empty or contains only whitespace should be treated as missing.
Field
	Type
	Regex
	Notes
	first_name
	str, required
	^[A-Za-z .,'-]{1,30}$
	Letters plus spaces, periods, commas, apostrophes, hyphens
	last_name
	str, required
	^[A-Za-z .,'-]{1,30}$
	Same character set as first name
	ssn
	str, required, unique
	^\d{3}-\d{2}-\d{4}$
	e.g. 123-45-6789
	email
	str, required, unique
	^[A-Za-z0-9]+@[A-Za-z]+\.[A-Za-z]+$
	Max 50 characters
	street
	str, required
	^[A-Za-z0-9 .,'-]{1,30}$
	Alphanumeric plus common characters
	city
	str, required
	^[A-Za-z .,'-]{1,30}$
	Letters plus common characters
	state
	str, required
	^[A-Z]{2}$
	Must also be checked against a set of real US state abbreviations — a regex alone can't verify "is a real state"
	zip_code
	str, required
	^\d{5}(-\d{4})?$
	5-digit or ZIP+4
	age
	int, required
	n/a (numeric check, not regex)
	Whole number, 0–120 inclusive
	height
	int, optional
	n/a (numeric check, not regex)
	Whole number, 0–108 inclusive, inches
	weight
	int, optional
	n/a (numeric check, not regex)
	Whole number, 0–1400 inclusive, pounds
	gender
	str, required
	n/a (membership check)
	Must be exactly "Male", "Female", or "Other" (case-sensitive)
	insurance
	str, required
	^[A-Za-z .,'-]{1,50}$
	Letters plus common characters
	Notes on the fields above that aren't pure regex:
* age, height, and weight should be validated as numeric range checks, not string patterns — try/except an int() conversion, or check str.isdigit() before converting, then check the range.
* state needs a second check beyond the regex: maintain a set or tuple of valid two-letter state codes and confirm membership.
* gender needs a second check beyond "is it a string": confirm the value is one of the three allowed, exact-case options.
* For ssn and email uniqueness, check the value against every patient already stored in memory before accepting it.
Every error message should tell the user which field failed and why (required vs. bad format), so someone reading the output could fix their input without guessing. For example: "zip_code: must be a 5-digit code or 5+4 format, e.g. 12345 or 12345-1234".

Non-Functional Requirements
1. README includes, at minimum, a description, prerequisites, and usage section (how to run the program from the command line), each under its own header.
2. Code follows PEP 8 style conventions.
3. Every function and class has a docstring describing what it does, its parameters, and its return value.
4. Type hints are used on all function signatures and dataclass fields.
5. No linting errors (e.g. via flake8 or pylint — pick one and note it in the README along with how to run it).
6. Unit tests (using pytest or unittest) cover the validators — both valid and invalid cases for every field — and reach at least 75% line coverage of the validation module. Include instructions in the README for how to run the tests with coverage (e.g. pytest --cov).
7. Validation logic lives in its own module, separate from the console menu/input-handling code, so it can be unit tested without mocking input().
